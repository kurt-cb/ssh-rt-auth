# ssh-rt-auth — installation guide

**Audience:** operators standing up an ssh-rt-auth deployment for the
first time. Focuses on:

- The **CA cert-issue flow** end-to-end (what gets signed by what, who
  holds which key).
- **Secure storage** of every long-lived secret. There are several;
  each has a different blast radius if compromised.
- Day-1 / day-2 operational hygiene.

This document does **not** cover the wrapper code itself yet — that's
the next-phase deliverable in [wrapper/](wrapper/). The PoC's
implemented surface today is **Tier 2** (the AsyncSSH server in
[server/](server/)) and **Tier 3** (the OpenSSH `AuthorizedKeysCommand`
shim in [openssh/](openssh/)).

---

## 1. Trust roots overview

ssh-rt-auth has **three independent cryptographic trust roots**. Mixing
them up is the most common operator mistake.

| Trust root                  | Key generated by              | Purpose                                          | Signs what                                                |
|-----------------------------|-------------------------------|--------------------------------------------------|-----------------------------------------------------------|
| **Authorization-signing key** | `ssh-rt-admin init` on the CA | Signs every short-lived authz cert returned to a shim | The X.509 authorization certs with policy extensions. |
| **mTLS root CA key**        | `ssh-rt-admin init` on the CA | Signs all mTLS server / admin / shim client certs | The CA's TLS server cert, the bootstrap admin client cert, every enrolled server's client cert. |
| **(Wrapper-only, future) Local user-CA key** | `ssh-rt-wrapper-admin init` on each wrapper host | Signs ephemeral OpenSSH user certs handed to the hermetic inner sshd | Per-connection 30-second OpenSSH user certs. **Never leaves its host.** Each wrapper host has its OWN local user-CA. |

The first two are issued *once*, per ssh-rt-auth deployment. The third
(if/when you deploy the wrapper) is issued *per host* — there's no
shared global "wrapper user-CA". This is intentional: it limits the
blast radius of a single compromised wrapper host.

## 2. CA bootstrap — what `ssh-rt-admin init` actually does

`ssh-rt-admin init --ca-dir /etc/ssh-rt-auth/ca` runs the
[`bootstrap_ca()`](ca/cert_minter.py) function. It produces **eight
artifacts** in `/etc/ssh-rt-auth/ca/`:

| File                          | What it is                          | Sensitivity   | Recommended mode |
|-------------------------------|-------------------------------------|---------------|------------------|
| `signing-key.pem`             | Authorization-signing **PRIVATE** key  | **CRITICAL**  | `0600` root:root |
| `signing-cert.pem`            | Self-signed cert wrapping the auth signing key (PUBLIC) | Public — distributed to every shim as `auth_trust_root` | `0644` |
| `tls-ca-key.pem`              | mTLS root CA **PRIVATE** key           | **CRITICAL**  | `0600` root:root |
| `tls-ca-cert.pem`             | mTLS root CA cert (PUBLIC)             | Public — distributed to every shim as `ca_trust_root` | `0644` |
| `tls-server-key.pem`          | CA's TLS server **PRIVATE** key        | sensitive     | `0600` root:root |
| `tls-server-cert.pem`         | CA's TLS server cert (PUBLIC)          | public        | `0644` |
| `bootstrap-admin-key.pem`     | Bootstrap superuser client **PRIVATE** key | sensitive     | `0600` admin user |
| `bootstrap-admin-cert.pem`    | Bootstrap superuser client cert        | public        | `0644` |

### 2.1 Why two private keys at the root?

- The **authorization-signing key** is consulted on *every single SSH
  connection* (via the CA's `POST /v1/authorize` endpoint). It signs
  short-lived X.509 authz certs that travel to shims and the wrapper.
- The **mTLS root CA key** is consulted only at `admin server add` /
  `admin user add` time, when minting an enrolment cert for a new
  server or admin. After enrollment, it sits idle.

Keeping them separate means:

- Day-to-day load is on the auth-signing key only.
- The mTLS root CA key can be kept **offline** after enrollments are
  done — see § 6.3.

## 3. The per-request CA cert-issue flow

This is the hot path every shim/wrapper takes on every SSH
authorization. Read this if you ever debug a deny.

```
Client SSH connection
    │
    │  (ssh wire-format pubkey blob + source IP/port)
    │
Server (running shim / wrapper)
    │  mTLS to one of ca.endpoints
    │  presents its enrolled server cert (signed by mTLS root)
    ▼
CA — Flask, mTLS listener (ca/server.py)
    │
    │  ┌─────────────────────────────────────────────────────────┐
    │  │ 1. extract server_subject from peer mTLS cert            │
    │  │    [ca/authorize.py]                                     │
    │  │    if not in enrollment DB → 401                         │
    │  │                                                          │
    │  │ 2. parse identity blob                                   │
    │  │    [ca/identity_parser.py]                               │
    │  │    SHA-256 fingerprint → look up enrolled user           │
    │  │    if no user → 403 unknown_identity                     │
    │  │                                                          │
    │  │ 3. evaluate policies                                     │
    │  │    [ca/policy.py: evaluate_policies()]                   │
    │  │    server match × user match × time-window × source-CIDR │
    │  │    if no matching policy → 403 no_matching_policy        │
    │  │                                                          │
    │  │ 4. mint authz cert                                       │
    │  │    [ca/cert_minter.py: mint_authorization_cert()]        │
    │  │    builds X.509 with critical extensions at the          │
    │  │    1.3.6.1.4.1.55555.1.{1..7} OID arc:                   │
    │  │      .1 source-bind     .2 server-bind                   │
    │  │      .3 channel-policy  .4 force-command                 │
    │  │      .5 environment     .6 max-session                   │
    │  │      .7 2fa-evidence                                     │
    │  │    SIGNED BY signing-key.pem (the auth-signing root)     │
    │  │                                                          │
    │  │ 5. audit log entry                                       │
    │  │    [ca/audit.py]                                         │
    │  │    JSON-lines: decision, serial, identity, server,       │
    │  │    connection, policy_applied                            │
    │  │                                                          │
    │  │ 6. response                                              │
    │  │    200 granted: {cert: base64-DER, serial, not_after}    │
    │  │    or 4xx denied with structured reason                  │
    │  └─────────────────────────────────────────────────────────┘
    │
    ▼
Server (back in shim / wrapper)
    │
    │  ┌─────────────────────────────────────────────────────────┐
    │  │ 7. validate cert (defense in depth)                      │
    │  │    [shim/shim.py: _validate_cert()]                      │
    │  │    - signature verifies against auth_trust_root          │
    │  │    - cert.not_after > now                                │
    │  │    - every critical extension OID in KNOWN_CRITICAL      │
    │  │                                                          │
    │  │ 8. cache + return                                        │
    │  │    [shim/cache.py or shim/sqlite_cache.py]               │
    │  │    keyed on (fingerprint, source_ip), TTL ≤ cert         │
    │  │    not_after                                             │
    │  └─────────────────────────────────────────────────────────┘
    ▼
    SSH session proceeds (or auth fails)
```

### 3.1 Where the keys actually live during this flow

| Step | Key used                                    | Where it sits           |
|------|---------------------------------------------|-------------------------|
| 0    | Shim's **mTLS client key** (server's cert)   | shim host disk, mode 0600 |
| 1    | CA's **mTLS server key**                     | CA host disk, mode 0600   |
| 1    | CA's **mTLS CA cert** (verifies the shim)    | CA host disk, mode 0644   |
| 4    | CA's **authorization-signing key**           | CA host disk, mode 0600 — read on EVERY request |
| 7    | Shim's **auth-trust-root cert** (verifies cert sig) | shim host disk, mode 0644 |

The auth-signing key reading on every request is by design — but it's
the reason that key MUST live on the box that runs the CA daemon, and
not on a separate file server or HSM that adds round-trips. (Future
HSM support can amortize this with caching; out of PoC scope.)

## 4. Installing the CA

### 4.1 System requirements

- Linux (any modern distro; the PoC's reference target is Alpine with
  Python).
- Python ≥ 3.9.
- ~50 MB disk for the CA dir + audit logs (grows ~1 KB/decision).
- Network reachability from every server-side shim **to the CA only**.
  The CA does **not** call out — it's strictly request-response.

### 4.2 Install

```bash
sudo useradd -r -s /usr/sbin/nologin ssh-rt-auth-ca   # service account
sudo mkdir -p /etc/ssh-rt-auth/ca /var/log/ssh-rt-auth /var/lib/ssh-rt-auth
sudo chown ssh-rt-auth-ca:ssh-rt-auth-ca /etc/ssh-rt-auth/ca \
    /var/log/ssh-rt-auth /var/lib/ssh-rt-auth
sudo chmod 0750 /etc/ssh-rt-auth/ca

# install the project (Python wheel / pip install / git clone — your call)
sudo pip install -r requirements.txt

# bootstrap
sudo -u ssh-rt-auth-ca ssh-rt-admin init \
    --ca-dir /etc/ssh-rt-auth/ca \
    --tls-server-san "DNS:ca.internal,IP:10.0.0.10"
```

`init` will print the bootstrap admin's mTLS cert + key paths.
**Take an offline copy of those two files now** — they are how you
recover from a "lost all admins" scenario.

### 4.3 Write `ca-config.yaml`

```yaml
listen: 0.0.0.0:8443
signing_key:     /etc/ssh-rt-auth/ca/signing-key.pem
signing_cert:    /etc/ssh-rt-auth/ca/signing-cert.pem
tls_cert:        /etc/ssh-rt-auth/ca/tls-server-cert.pem
tls_key:         /etc/ssh-rt-auth/ca/tls-server-key.pem
client_ca_cert:  /etc/ssh-rt-auth/ca/tls-ca-cert.pem
identity_trust_roots: []          # populate with ssh user-cert CAs if used
enrollment:
  path: /etc/ssh-rt-auth/ca/enrollment.yaml
audit:
  path: /var/log/ssh-rt-auth/audit.jsonl
defaults:
  max_cert_validity_seconds: 3600
  timestamp_drift_seconds:   600
cert_generation:
  key_type: ec                    # or ed25519
  server_cert_validity_days: 30
  admin_cert_validity_days:  30
```

### 4.4 Run

```bash
sudo -u ssh-rt-auth-ca python3 -m ca.server --config /etc/ssh-rt-auth/ca-config.yaml
```

Or via a systemd unit; example in [config/](config/).

## 5. Enrolling a server

Each SSH server (Tier 2 or Tier 3) needs:

1. A canonical name (e.g. `web-prod-01`).
2. An mTLS client cert+key signed by the mTLS root, distributed to
   `/etc/ssh-rt-auth/shim/mtls-cert.pem` and `mtls-key.pem` on that
   host.
3. The CA's `signing-cert.pem` (the **auth-trust-root**) distributed
   to that host as `/etc/ssh-rt-auth/shim/auth-trust-root.pem`.
4. The CA's `tls-ca-cert.pem` (the **mTLS-trust-root**) distributed
   as `/etc/ssh-rt-auth/shim/ca-trust-root.pem`.

```bash
# On the CA host, as the bootstrap admin:
ssh-rt-admin server add \
    --name web-prod-01 \
    --groups production,web \
    --output-dir /tmp/web-prod-01-enrollment

# This produces:
#   /tmp/web-prod-01-enrollment/mtls-cert.pem
#   /tmp/web-prod-01-enrollment/mtls-key.pem
#   /tmp/web-prod-01-enrollment/auth-trust-root.pem  (copy of signing-cert.pem)
#   /tmp/web-prod-01-enrollment/ca-trust-root.pem    (copy of tls-ca-cert.pem)

# SCP the directory to the server (over a trusted channel) and install.
scp -r /tmp/web-prod-01-enrollment web-prod-01:/etc/ssh-rt-auth/shim/
# Then on web-prod-01:
sudo chmod 0600 /etc/ssh-rt-auth/shim/mtls-key.pem
sudo chmod 0644 /etc/ssh-rt-auth/shim/*.pem
sudo rm /tmp/web-prod-01-enrollment   # don't leave the key in /tmp
```

## 6. Secure storage — guidance per key

This is the most important section in this doc.

### 6.1 Authorization-signing key (`signing-key.pem`)

**Sensitivity:** CRITICAL. Compromise = attacker can mint authz certs
for any user/server. Catastrophic.

- **Owner:** the CA service account.
- **Mode:** `0600`.
- **Location:** CA host disk, in `/etc/ssh-rt-auth/ca/`. Treat the host
  as a **management-tier asset** — same access controls as a Kerberos
  KDC or your secrets manager.
- **Backup:** offline, encrypted (gpg or a sealed FIPS device).
  Two-person rule recommended.
- **Rotation:** annually, or immediately on suspected compromise. Use
  `ssh-rt-admin rotate-signing-key`. Shims pick up the new
  auth-trust-root via the next config push; old certs continue to
  validate until rotation completes.
- **Do not** put this key on a Tier 3 host. Do not put it in your CI
  system. Do not put it in a container image. Do not check it into
  source control. Do not generate it on a laptop.

### 6.2 mTLS root CA key (`tls-ca-key.pem`)

**Sensitivity:** CRITICAL when used; idle most of the time.

- Same on-disk treatment as the auth-signing key.
- **STRONGLY RECOMMENDED:** after initial enrollments are complete,
  **take this key offline.** Move it to an encrypted USB drive, store
  it in a safe, remove it from the CA host. Re-mount it only when
  enrolling a new server or admin. The CA daemon does not need it at
  runtime (it only uses `tls-server-key.pem` for its own mTLS server
  identity, which is a leaf cert signed by this root).
- Rotation: every 2-3 years or on suspected compromise. Coordinated
  re-enrollment of all servers.

### 6.3 CA's TLS server key (`tls-server-key.pem`)

**Sensitivity:** moderate. Compromise = attacker can impersonate the
CA on the network *if* they also control the IP/DNS, which is a
high-effort attack inside a private network.

- `0600` root:root.
- Auto-rotated by re-running `ssh-rt-admin renew-tls-server-cert`
  every ~30 days (you can wire this into cron).

### 6.4 Server mTLS client key (`mtls-key.pem`)

**Sensitivity:** moderate. Compromise = attacker can talk to the CA
*as that server* — they could request authz certs for any user the
server's policies allow. But they still can't bypass *user-side*
authentication unless they also hold the user's SSH private key. And
they can only do this from a place that can reach the CA's mTLS
endpoint (private network only).

- `0600` owned by the shim service account.
- Per-server, never shared.
- Rotation: every 30-90 days via `ssh-rt-admin server rotate-key
  --name web-prod-01`.

### 6.5 Bootstrap admin key (`bootstrap-admin-key.pem`)

**Sensitivity:** CRITICAL on first boot, then **delete it**.

- The bootstrap admin exists only to enroll the *real* admin users.
- After your first non-bootstrap superuser admin is enrolled and
  verified working, **revoke the bootstrap admin** (`ssh-rt-admin
  admin remove bootstrap-admin`) and **delete the key file from the
  CA host**.
- If you ever need to recover: the offline copy you took in §4.2 is
  the only path back. Keep that copy somewhere your incident-response
  team can reach.

### 6.6 (Future, wrapper) Local user-CA key per wrapper host

**Not yet shipped — wrapper implementation in progress.**

When the wrapper ships:

- Each wrapper host generates its own Ed25519 user-CA on `ssh-rt-wrapper-admin
  init`. **Stored only on that host**, mode `0600`.
- The wrapper holds it **in memory** after startup; reads from disk
  only on SIGHUP.
- **Not backed up.** If lost, run `init` again — the only consequence
  is that the wrapper's inner sshd gets a new `TrustedUserCAKeys`
  pubkey, transparent to end users.
- Compromise of one wrapper host's user-CA does **not** affect any
  other host, the CA, or other users. Blast radius = that one host.

## 7. Day-2 operations

### 7.1 Audit-log retention

`/var/log/ssh-rt-auth/audit.jsonl` grows ~1 KB per authorization
decision. At 10 logins/second that's ~80 MB/day uncompressed, ~8 MB/day
gzipped.

- Logrotate config:

  ```
  /var/log/ssh-rt-auth/audit.jsonl {
      daily
      rotate 90
      compress
      delaycompress
      missingok
      postrotate
          /bin/kill -HUP $(cat /var/run/ssh-rt-auth-ca.pid)
      endscript
  }
  ```

- Ship to your SIEM. The schema is documented in
  [docs/ssh-rt-auth-detailed-ca-admin.md](docs/ssh-rt-auth-detailed-ca-admin.md).

### 7.2 Monitoring

Minimum metrics to alert on:

- CA process is up.
- CA `POST /v1/authorize` 5xx rate.
- Audit-log writes (failure to write = silent denials in the future).
- Disk space on `/var/log/ssh-rt-auth/`.
- Cert expiry: every server's mTLS client cert ≥ 14 days from
  expiring.

### 7.3 Disaster recovery — "we lost everything"

In rough order of severity:

| Lost                              | Recovery                                                                                                              |
|-----------------------------------|------------------------------------------------------------------------------------------------------------------------|
| One shim host                     | Re-enroll with `ssh-rt-admin server add`. Existing authz certs the host had cached become useless; new ones are fine. |
| One wrapper's local user-CA       | `ssh-rt-wrapper-admin init` on that host. Auto-self-heals.                                                            |
| All shim hosts                    | Same — each is independent.                                                                                            |
| CA process / disk (keys intact)   | Restore from backup. Service resumes; no client-side change.                                                          |
| Auth-signing key                  | If you have the offline backup: restore + verify shims still validate. If you don't: full reset (issue new auth-signing key, force every shim to re-pick up the new auth-trust-root). |
| mTLS root CA key                  | If offline backup intact: restore. If not: re-bootstrap + re-enroll EVERY server (painful). |
| Bootstrap admin key (after step §6.5) | This was supposed to be deleted. If a real superuser still exists, they can enroll a replacement. If not, full re-bootstrap. |

### 7.4 Time skew — the CA is the authoritative clock

Authz certs have tight validity windows (~1 hour by default). A wrapper
or shim whose local clock disagrees with the CA's by more than
`defaults.timestamp_drift_seconds` (default 600s) will see every
authorization request rejected with `clock_drift_too_large` — a
silent, fleet-wide DOS that's hard to diagnose if you don't know to
look for it.

**Architecture: the CA is the time authority, not the operator's NTP
mesh.** The wrapper synchronises its time perception to the CA at
startup and periodically thereafter, then uses *CA-derived time* for
every timestamp it sends to the CA. This eliminates the entire class
of "bad client clock causes auth failures" — there is, by design, no
way for a client's local clock to be wrong from the CA's perspective.

```
  Operator → ensure ONE thing has correct time: the CA host.
                              │
                              ▼
  Wrapper → calls /v1/clock at startup; stores offset.
            Every CA request uses CA-derived time.
            Re-syncs every 5 minutes in background.
            Refuses to start if CA unreachable
            (fail-closed: a broken offset = silent denials).
                              │
                              ▼
  mssh client → checks local skew against /v1/clock at start.
                Errors out loudly ("your clock is 17 hours off") if
                skew exceeds a safe threshold. Does NOT auto-bypass —
                the client must have working time for mTLS dates.
```

**What this means for you, the operator:**

1. **Run NTP/chrony on the CA host with multiple upstream sources.**
   Use `chronyd` with at least 3 upstream NTP servers (e.g., your
   org's NTP cluster plus `2.pool.ntp.org`). The CA host is the
   single authoritative clock for ssh-rt-auth; if it drifts, the
   whole fleet drifts with it.
2. **Monitor the CA host's clock health.** `chronyc tracking` should
   show `Last offset` and `Root dispersion` both below 100 ms. Alert
   on either exceeding 500 ms.
3. **On every other host (wrappers, Tier 2/3 servers), NTP is
   nice-to-have but not load-bearing for ssh-rt-auth.** The wrapper
   syncs to the CA at the app layer. The mssh client warns the user
   if their clock is bad. You don't need to run NTP on every laptop.
4. **CA outage = wrapper denies new authorizations** (as before),
   but additionally: a long CA outage stales the wrapper's offset.
   The wrapper enters a "degraded clock" state after 1 hour of
   missed re-syncs and refuses to mint inner OpenSSH certs.
   Surfaced as `ssh-rt-wrapper-admin status` field
   `clock_sync_state`.

**Failure modes:**

| Bad clock on…           | Behaviour                                                                    |
|-------------------------|------------------------------------------------------------------------------|
| CA host                 | **Catastrophic.** Whole fleet sees auth failures. ONE thing to monitor — that's the point. |
| Wrapper host            | Mitigated. Wrapper applies CA-derived offset.                                  |
| Tier 2/3 server host    | Mitigated (Tier 2/3 shims also adopt the same offset mechanism in Phase 1B+). |
| End-user laptop         | mssh CLI errors out at start with a clear message; user sets their clock and retries. |
| Inner sshd (same host as wrapper) | Irrelevant — wrapper and inner sshd share a clock; CA never enters the loop. |

**Implementation status:** documented design as of 2026-05-14. The CA
endpoint and wrapper Clock module are not yet implemented. See
[design/ssh-rt-auth-detailed-wrapper.md § 6.6](design/ssh-rt-auth-detailed-wrapper.md)
for the spec.

## 8. Hardening checklist

- [ ] CA host is a dedicated VM/container; nothing else runs on it.
- [ ] CA host firewall: only the mTLS listen port is reachable from
      the shim subnet; no outbound.
- [ ] CA `signing-key.pem` is mode 0600, owned by the CA service user.
- [ ] `tls-ca-key.pem` is taken **offline** after initial enrollments.
- [ ] Bootstrap admin is removed and its key deleted from disk; offline
      copy is in your recovery vault.
- [ ] Every server's `mtls-key.pem` is mode 0600.
- [ ] Logrotate is configured and the SIEM is consuming
      `audit.jsonl`.
- [ ] NTP/chrony is running on the CA and every shim host.
- [ ] You have tested disaster recovery for at least the "single shim
      host lost" and "CA key offline backup recovery" cases.
- [ ] Cert expiry alerts are wired up.
